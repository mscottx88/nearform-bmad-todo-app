# Story 5.2: Hybrid Search API

Status: review

> **Scope note:** Second story of Epic 5 (Intelligent Search). Builds directly on Story 5.1's embedding pipeline: this story adds a new `GET /api/search` endpoint that combines PostgreSQL full-text search (GIN index on `to_tsvector('english', text)`) with pgvector cosine-similarity search (HNSW index on `embedding VECTOR(768)`). Both indexes landed in the initial migration at [backend/migrations/versions/7af34c6df37c_initial_schema.py:40-52](backend/migrations/versions/7af34c6df37c_initial_schema.py#L40-L52). Query-side embedding generation reuses `embedding_service.generate_embedding` from Story 5.1. Frontend integration (type-anywhere search UI, surface/submerge animation, 300 ms debounce) lands in Story 5.3.

## ⚠️ CRITICAL CONSTITUTIONAL CONSTRAINT

**Async/await is PROHIBITED in this codebase** — see [CLAUDE.md](CLAUDE.md) § "CONCURRENCY MODEL — THREAD-BASED ONLY".

The search endpoint:

- ❌ **NEVER** `async def`, `await`, `asyncio`, FastAPI `BackgroundTasks`, `aiohttp`, `httpx` async client, `asyncpg`.
- ✅ Sync FastAPI route handler (`def search(...)`), synchronous `psycopg` via `SQLAlchemy`/`Session`, `ThreadPoolExecutor` if parallelism is needed.

Same policy as Story 5.1 — the existing `embedding_service.generate_embedding` is already sync and blocks; that's fine for the query path because the user IS actively waiting. The search's own latency budget (≤300 ms per NFR) is what constrains us, not a threading choice.

## Story

As a user (via the future Story 5.3 search UI),
I want a single endpoint that finds todos whose text or meaning matches my query,
So that typing "docs" surfaces my "update README" todo (semantic) and "Review Q2" (keyword) together, ranked by combined relevance, with graceful degradation to keyword-only when the embedding API is down.

## Acceptance Criteria

1. **Given** the backend has active todos (some with `embedding_status='complete'`, some `'pending'`, some `'failed'`), **When** a client calls `GET /api/search?q=review`, **Then** the response is a JSON object with shape `{"query": "review", "results": [{"todo": {...TodoResponse...}, "score": <float 0..1>, "match_type": "keyword" | "semantic" | "hybrid"}, ...], "vector_search_unavailable": false}`. `results` is sorted by `score` descending. The `todo` object is the same `TodoResponse` shape used by `GET /api/todos`. `match_type` reports which side of the hybrid produced the hit: `"keyword"` = FTS only, `"semantic"` = vector only, `"hybrid"` = both. Frontend (Story 5.3) will use this to decide whether to highlight matching substrings in the pad text.

2. **Given** a valid non-empty `q`, **When** the endpoint runs, **Then** it executes TWO SQL queries and merges in Python:
   (a) **Full-text query** using `websearch_to_tsquery('english', :q)` matched against the existing `ix_todos_text_search` GIN index; projected score is `ts_rank_cd(to_tsvector('english', text), websearch_to_tsquery('english', :q))` normalised to `[0, 1]` via `ts_rank_cd / (1 + ts_rank_cd)`.
   (b) **Vector query** using `1 - (embedding <=> :query_vec)` (cosine similarity; `<=>` is pgvector's cosine distance operator matching the HNSW `vector_cosine_ops` index) — only rows where `embedding IS NOT NULL AND embedding_status = 'complete'` participate. The query embedding is generated by calling `embedding_service.generate_embedding(q)`.
   Both queries filter on `deleted = false AND completed = false` per **FR14** ("completed and deleted todos are excluded from results"). Archived is currently always false but include `archived = false` for forward-compat. Each query `LIMIT`s to `MAX_CANDIDATES_PER_SIDE = 50`.

3. **Given** the two candidate sets from AC #2, **When** merging, **Then** the final score for each unique `todo.id` is:
   `score = FTS_WEIGHT * fts_score_normalised + VECTOR_WEIGHT * vector_similarity` where `FTS_WEIGHT = 0.3`, `VECTOR_WEIGHT = 0.7` (constants in `search_service`; documented rationale: semantic matches weighted higher than keyword hits, but keyword hits are never zeroed out so an exact-substring match always ranks above a weakly-related semantic match). Todos present in only one result set use `0.0` for the missing score's term (not the missing weight — e.g., a row with only a FTS hit ranks as `0.3 * fts_normalised`, giving a ceiling of 0.3 for FTS-only results). Sort by final score desc; tie-break by `created_at` desc. Return top `RESULT_LIMIT = 20`.

4. **Given** `embedding_service.generate_embedding(q)` raises any exception (missing API key, HTTP timeout, `ClientError` 4xx/5xx, malformed response, `EmbeddingDimensionError`), **When** the endpoint handles it, **Then** it logs a single WARNING with the exception type (and `.code`/`.status` if present, reusing the pattern from `embedding_worker`'s enriched log) and falls back to **FTS-only results** with `vector_search_unavailable: true` in the response. The response is still HTTP 200. No retries on the search path — retries here would exceed the 300 ms UX budget. (Contrast: the Story 5.1 background worker DOES retry with backoff; the search path does not.)

5. **Given** `q` is empty or whitespace-only (after `.strip()`), **When** the endpoint receives it, **Then** it returns HTTP 422 with the standard `validation_error` response body (message: "Query cannot be empty"). `q` with `len > 500` chars is also rejected with 422 (message: "Query too long") — this guards the Google embedding API's input budget and makes the public surface slightly harder to abuse.

6. **Given** the request URL has no `q` parameter at all (`GET /api/search`), **When** FastAPI/pydantic validates the request, **Then** the response is HTTP 422 with a `validation_error` body. `q` is a **required** query parameter (not optional with a default).

7. **Given** the search endpoint is a sync `def search(...)` route, **When** requests arrive, **Then** the handler uses the request's `Session` via the `get_db` dependency (does NOT open its own `SessionLocal()` — that's a worker-thread pattern, not a request-handler pattern). All SQL goes through parameterised SQLAlchemy `text()` constructs or Core `select()` statements — **NO** raw string interpolation of `q` into SQL (PostgreSQL's `websearch_to_tsquery` already handles untrusted input safely, but parameter-binding is the safety net).

8. **Given** the query embedding is a `list[float]` of length 768, **When** it's used in the pgvector cosine-distance query, **Then** it is passed as a parameter to the SQL — **not** string-formatted into the query. The pgvector SQLAlchemy `Vector` adapter handles binding; using `bindparam(..., type_=Vector(768))` is the required idiom.

9. **Given** the new search module is introduced, **When** the existing pytest suite + this story's new tests run, **Then** (a) all existing backend tests still pass (no regressions in `test_todos.py`, `test_todo_service.py`, `test_embedding_*` tests — 43 tests as of Story 5.1 close), (b) a new `backend/tests/api/test_search.py` covers: missing-q (422), empty-q (422), too-long-q (422), q-with-only-keyword-matches-returns-FTS-only-results, q-with-only-semantic-matches-returns-vector-results, mixed-matches-returns-combined-ranked, Google-API-failure-falls-back-to-FTS-only (response `vector_search_unavailable: true`), deleted-todos-excluded, completed-todos-excluded, pending-embedding-todos-only-FTS-searchable (no vector contribution), score-ordering-correctness (manually constructed fixture whose FTS+vector scores are computable by hand to within a small epsilon).

10. **Given** the endpoint is registered and documented, **When** FastAPI generates OpenAPI docs, **Then** `/api/search` appears under a new `search` tag with the correct response schema. Router is included in `main.py` alongside `todos_router` and `creatures_router`.

11. **Given** the search path includes a network call (`generate_embedding` over HTTPS) inside a sync request handler, **When** a stuck Google API would otherwise pin the request thread, **Then** the existing `HttpOptions(timeout=15_000 ms)` from Story 5.1 bounds the worst-case request. No new timeout plumbing is required in this story. (The 15 s ceiling is ugly for a 300 ms NFR budget, but the story-5.1 timeout exists specifically to prevent unbounded blocking; a tighter search-path timeout is a defer item — see Dev Notes § Out of scope.)

## Dev Notes

### Hybrid-scoring algorithm rationale

**Why FTS=0.3 / vector=0.7?**

PostgreSQL `ts_rank_cd` scores are dense-frequency ranks that depend on document length + query-term weights; they're not naturally bounded to `[0, 1]`. The `ts_rank_cd / (1 + ts_rank_cd)` transform is a common "squashing" function that maps `[0, ∞)` to `[0, 1)` monotonically. Values above ~0.5 on this scale indicate a very strong keyword match (multiple exact hits in a short document).

Cosine similarity on normalised embeddings is naturally in `[-1, 1]`; for sensibly generated embeddings of short todo text, the useful range clusters around `[0.2, 0.95]`. `similarity = 1 - cosine_distance` maps pgvector's `<=>` operator output directly.

A weighted sum `0.3 * FTS + 0.7 * vec` means:
- A pure-keyword hit tops out at `0.3` (the FTS weight) — so "Review roadmap" → `q="review"` → score ≈ 0.25, still a clear winner over unrelated todos.
- A pure-semantic hit with high similarity (0.9) tops out at `0.63`.
- A dual hit can reach `0.95+`.

This keeps the common case "I typed a word that's literally in the todo" ranking well while letting semantic-only matches show up when the user types concept-level queries ("docs" → "update README").

Make these constants, not settings — tuning them without a reproducible eval dataset is guesswork. Revisit only when we have real usage data.

### SQL patterns — copy these, don't invent

**FTS (SQLAlchemy Core with textual SQL for the `@@` operator):**

```python
from sqlalchemy import text

stmt = text("""
    SELECT id, text, completed, color, position_x, position_y,
           embedding_status, archived, archived_at, deleted, deleted_at,
           created_at, updated_at,
           ts_rank_cd(
               to_tsvector('english', text),
               websearch_to_tsquery('english', :q)
           ) AS fts_score
    FROM todos
    WHERE to_tsvector('english', text) @@ websearch_to_tsquery('english', :q)
      AND deleted = false
      AND completed = false
      AND archived = false
    ORDER BY fts_score DESC
    LIMIT :max_candidates
""")
rows = db.execute(stmt, {"q": q, "max_candidates": MAX_CANDIDATES_PER_SIDE}).all()
# Materialise as (Todo ORM instance, score) via Todo(**row._mapping) — or
# run a second ORM query by id; either works.
```

Reality check: pulling raw rows and hand-constructing `Todo` ORM objects is brittle (new columns would break this). **Preferred alternative**: select ids + fts_score from SQL, then `db.query(Todo).filter(Todo.id.in_(ids))` to get hydrated ORM objects, then stitch the scores back via a dict by id. Dev agent should use the preferred alternative.

**Vector (pgvector operator `<=>`):**

```python
from sqlalchemy import bindparam, text
from pgvector.sqlalchemy import Vector

stmt = text("""
    SELECT id,
           1 - (embedding <=> :query_vec) AS similarity
    FROM todos
    WHERE embedding IS NOT NULL
      AND embedding_status = 'complete'
      AND deleted = false
      AND completed = false
      AND archived = false
    ORDER BY embedding <=> :query_vec
    LIMIT :max_candidates
""").bindparams(bindparam("query_vec", type_=Vector(768)))

rows = db.execute(
    stmt,
    {"query_vec": query_vec, "max_candidates": MAX_CANDIDATES_PER_SIDE},
).all()
ids = [r.id for r in rows]
id_to_sim = {r.id: r.similarity for r in rows}
todos = db.query(Todo).filter(Todo.id.in_(ids)).all() if ids else []
```

The `embedding <=> :query_vec` form (with the operator in both `SELECT` and `ORDER BY`) is what triggers HNSW index usage — pgvector's query planner needs the distance operator in `ORDER BY` to pick the index. Don't rewrite this as `ORDER BY similarity DESC` on the aliased column; that will sequential-scan.

### Graceful degradation — exactly one retry policy

This story explicitly does **NOT** retry the query embedding. Rationale:

- The user is waiting synchronously (search UI is real-time / 300 ms debounced per Story 5.3).
- A retry + backoff would easily blow past 1 s, making the UI feel broken.
- FTS-only results are a totally usable fallback — keyword matches cover 80% of practical queries.
- The background-worker's retry logic in Story 5.1 is separate and unchanged; that's for durable writes (each todo gets embedded eventually), not for read-path freshness.

Log the failure with type + code + status (same enriched format as story 5.1's `embedding_attempt_failed` line). Operators can correlate search failures with worker failures.

### `Session` lifetime

The endpoint uses the request's `Session` via `Depends(get_db)` — do NOT open a new `SessionLocal()` like the background worker does. The worker's fresh-session pattern is because the worker runs in a different thread after the request's session has been closed. The search endpoint runs IN the request thread; reusing the request session is the correct pattern and keeps the query inside the request's transaction.

### Why `websearch_to_tsquery` and not `plainto_tsquery`?

- `plainto_tsquery` treats the entire input as AND of terms, no operator support.
- `websearch_to_tsquery` (Postgres ≥ 11) handles quoted phrases ("exact match"), `OR`, and `-word` exclusion — the Google-style query DSL users expect. It safely handles malformed input (unbalanced quotes, stray operators) by treating them as literals.
- For Story 5.3's type-anywhere UX, `websearch_to_tsquery` is strictly better at user-facing behaviour.

### FR14 edge case: completed-and-then-resurrected

The PRD says completion is terminal (no uncomplete path). The existing `list_todos` filter at [backend/src/services/todo_service.py:42-51](backend/src/services/todo_service.py#L42-L51) already excludes completed + deleted todos. Search must mirror this exclusion exactly.

Archived is filtered-out in the query even though `archived` is dead in v1 (see the schema comment `-- DEPRECATED: auto-archive removed in v1 simplification` in architecture.md). Including it in the filter is cheap insurance against future re-introduction.

### Deferred-work interaction

From Story 5.1's CR-defer list, two items are directly relevant:

- **"No reaper for stuck pending rows"**: this story doesn't fix it but does expose the consequence — a `pending` todo is only full-text-searchable here. Operationally, stuck rows become visible via "vector match count consistently < FTS match count" in logs. Logging that delta per-query would help operators spot the backlog, but is out of scope here.
- **"PATCH text doesn't re-embed"**: also out of scope; edited todos will have drift between text and embedding until the future defer is picked up. Search returns whichever matches best — users shouldn't hit surprising results in the v1 demo (text is rarely edited).

Neither deferred item blocks this story; both might get absorbed into Story 5.3 or a follow-up "search quality" ticket.

### Out of scope for 5.2

- **Tighter search-path timeout** — the existing 15 s `HttpOptions(timeout=15_000)` caps worst-case. A dedicated `HttpOptions` (~1.5 s) for the search path would improve the user experience during a slow Google API but is a config-plumbing change (separate clients or config-per-call). Defer until real latency data says it matters.
- **Pagination / offset** — `RESULT_LIMIT=20` is all the frontend needs for Story 5.3's surface/submerge animation (the pond displays ≤ ~30 pads at once in practice).
- **Filters / facets** — no date-range, color, or completion-status filters. Only active todos are searchable; that's the product spec.
- **Query logging / analytics** — no "top N searches" dashboard; no search-latency metric. Observability is deferred per architecture doc.
- **Re-rank with LLM** — combined FTS + vector is sufficient for small-scale search (≤ a few hundred todos). Proper Cross-Encoder re-ranking is 10× overkill and hits Google twice per query.
- **Typo correction / fuzzy FTS** — Postgres `trgm` extension would give this, but it's not installed. Users hitting typos get the semantic component to catch them (e.g., "revue" → reviews via embedding).
- **Frontend changes** — zero. Story 5.3 is where the pond learns to surface/submerge.

## Anti-Patterns to Avoid

- **DO NOT** make the route handler `async def`. Sync only. If you're typing `async`, stop and re-read CLAUDE.md § concurrency.
- **DO NOT** open a new `SessionLocal()` inside the route handler. Use `Depends(get_db)` — that's what the pattern is for.
- **DO NOT** string-format `q` or the query vector into SQL. Use `text(...)` with `bindparam` and parameter dicts. `websearch_to_tsquery` itself is injection-safe, but parameter binding is the safety net that catches a future refactor that adds a second concatenation.
- **DO NOT** sequential-scan the vector query. Keep `ORDER BY embedding <=> :query_vec` (operator in `ORDER BY`) to trigger HNSW. Don't alias and re-order.
- **DO NOT** call `generate_embedding` in a background thread on the search path. The user IS waiting — run synchronously. The 15 s timeout bounds worst case.
- **DO NOT** add retries to the search path. Story 5.1's worker retries; the search path falls back to FTS-only. Two different policies for two different places.
- **DO NOT** log `q`'s contents at WARNING/ERROR level. `q` may contain PII in future (e.g., a user searching for a co-worker's name). Log only the exception type + code + status on failure.
- **DO NOT** leak Google API key in error responses. Reuse Story 5.1's generic "service temporarily unavailable" pattern; here it surfaces as `vector_search_unavailable: true`, not an error.
- **DO NOT** expose scores > 1.0 or < 0.0. The normalisation must guarantee `[0, 1]`; assert this in the service or use `pydantic.Field(ge=0.0, le=1.0)` on `SearchResult.score` so a bug becomes a 500, not a corrupted client.
- **DO NOT** invent a new `SearchRequest` pydantic model just to validate `q`. FastAPI's `Query(..., min_length=1, max_length=500)` already does it cleanly.
- **DO NOT** use `plainto_tsquery`. Use `websearch_to_tsquery` for user-facing search.
- **DO NOT** break the existing HNSW index with a different operator. The index was created with `vector_cosine_ops` — the query MUST use `<=>` (cosine distance), not `<->` (L2) or `<#>` (inner product).

**Design decisions / deviations from the spec:**

1. **Validator exception type**: spec said "either `HTTPException(422)` OR pydantic validator — pick one." Chose the pydantic `AfterValidator` path so the response uses the existing `validation_error` envelope (same shape as "empty text on POST /api/todos"). Switched from `ValueError` to `PydanticCustomError` because a plain `ValueError` puts the exception instance into the error context, which then crashes the global `validation_error_handler`'s `exc.errors()` serialisation (not JSON-safe). `PydanticCustomError` keeps everything JSON-serialisable and doesn't require touching main.py's handler.
2. **Vector similarity clamp to `[0, 1]`**: the `<=>` operator returns cosine distance in `[0, 2]`; `1 - distance` can technically be negative for diametrically opposed vectors. Clamped so the `SearchResult.score` field constraint (`ge=0.0, le=1.0`) always holds and a pathological embedding can't turn into a 500.
3. **FTS candidate hydration**: the spec suggested either (a) stitching raw rows to `Todo` objects or (b) collecting IDs and re-querying via the ORM. Chose (b) as the spec's "preferred alternative" — a second `SELECT * FROM todos WHERE id IN (...)` is cheap and future-proof against new columns.
4. **Score combined clamp**: `max(0.0, min(1.0, combined))` in `_merge` defends against float-arithmetic drift (e.g., `0.7 + 0.3` in IEEE-754 can produce `1.0000000000000002`). Pydantic would otherwise reject the value.
5. **Extra test `test_hybrid_search_empty_result_set_returns_empty_list`** added beyond the spec's list to pin the "zero matches" happy path.
6. **Extra integration test `test_search_openapi_schema_includes_endpoint`** verifies AC #10's OpenAPI tag requirement programmatically.
